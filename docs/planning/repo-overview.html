Kolo · Repo Overview

A multiplatform platform baseline moving toward an app factory shape.

Kolo is a Kotlin Multiplatform baseline for future mobile apps with shared architecture, shared runtime infrastructure, and thin app shells. It is already beyond wizard-template territory, but it is not yet a finished end-user product app or a fully proven multi-app factory.

What it actually is today

A platform-oriented repo with Android and iOS reference hosts, shared KMP modules, shared build logic, internal tooling, and a small server starter. The platform foundation is real. Feature extraction and app-factory scaling are still in progress.

What comes next

The current sequence is persistence, navigation, reminder contracts, and the first real feature slices on top of the extracted store baseline, followed by deeper shared-module decomposition and more explicit app-shell scaling.

Module / Repo Map

What lives where

This is the live repo and module shape, not just aspirational architecture.
apps/reference/androidApp
The current Android host location for the real Gradle app module :androidApp. MainActivity is deliberately thin and mounts the shared UI.
apps/reference/iosApp
The current iOS host shell. It is a real repo host, but not a Gradle module.
composeApp
The shared Compose UI library. It is no longer the Android application module; it is the current shared presentation surface.
debugmenu
A separate KMP internal-tooling module for runtime config inspection and mutation, intentionally isolated from release-facing product UI.
shared
The remaining broad shared KMP library. It still contains app graph wiring and common runtime pieces outside the already extracted core modules.
shared:core:config:api / impl
The split runtime-config contract and implementation: pure models and read-side contracts in API, persistence / merge / mutation logic in impl.
shared:core:store:api / impl
The extracted shared store/UDF contract and runtime implementation used as the baseline state architecture.
baselineprofile
Android Baseline Profile generation. This is release-performance tooling, not a shipping app module.
server
A small Ktor server starter that currently exposes a minimal “Kolo server is running” endpoint.
build-logic
The included Gradle build that owns convention plugins and repo-wide build wiring. This is the source of truth for shared plugin stacks.
Runtime Path

How the app works today

This is the practical execution path, not just the intended architecture.

Android path

1
Android host
MainActivity enables edge-to-edge and renders KoloApp().
2
Shared UI
composeApp provides the current shared Compose-facing UI surface.
3
Shared graph
AndroidAppGraph provides the store factory and app config repository, plus Android-specific config mutation wiring.
4
Debug-only tooling
debugmenu is attached with debugImplementation, keeping the surface at the app edge instead of product UI.

iOS path

SwiftUI owns the host shell and window scene.
An iOS graph provides the same shared runtime contracts as Android, with iOS-specific local config persistence wiring.
The internal debug menu can be opened through a debug-only route and presented modally from the iOS shell.
Architectural intent Keep shells thin, keep reusable logic in shared Kotlin, and push platform-only bindings to the graph edge instead of leaking them through common code.
Roadmap

What happens next

This is the live planning sequence, not a speculative wishlist.

1. Persistence

Lock the v1 persistence model for tasks, journals, reminders, settings, and sync metadata.

2. Navigation

Define typed route ownership, back behavior, modal policy, deep links, restoration, and rollout sequencing.

3. Reminders & features

Formalize reminder scheduling contracts, then build the first real feature slices on the shared store baseline.
