Kolo · Implementation Cut

iOS now points at shared Compose, and the shared root is using Navigation 3 on top of a narrow app-shell contract.

This cut moves Kolo away from the split world where Android rendered shared Compose and iOS still showed a native placeholder. The app root is now shared Compose on both platforms at the source level, the shared root uses a typed Navigation 3 back stack instead of static placeholder content, and shell identity now comes from the extracted shared:core:appshell contract rather than the broader runtime-composition module.

Outcome

What changed

This is a foundation cut, not the full navigation architecture.
Platform move

iOS host now mounts Compose

The iOS app no longer stops at a SwiftUI placeholder screen. SwiftUI stays as a thin host, but the root UI is now provided by the shared Compose framework.

Shared root

Compose moved to commonMain

The main app composable and root presentation logic now live in shared code so Android and iOS point at the same app surface.

Shell contract

App identity is now narrow

The shared UI now reads shell identity and capability metadata from shared:core:appshell instead of depending on the broader shared runtime-composition module.

Navigation runtime

Navigation 3 is live

The shared root now owns a typed Navigation 3 back stack with serializable routes and a simple proof path through multiple destinations.

Current State

What the repo now supports

This is the structural answer to whether Kolo can actually carry shared multiplatform navigation.
Android root
Shared Compose
Still mounted from the Android host, now through the same commonMain root used elsewhere.
iOS root
Shared Compose
Mounted through a thin SwiftUI `UIViewControllerRepresentable` wrapper over `ComposeApp.framework`.
Navigation runtime
Navigation 3
Shared typed routes and saved-state configuration are now part of the root app surface.
Implementation Shape

What was added

The goal was the smallest coherent cut, not a broad rewrite.
Shared Compose

New root structure

`composeApp/src/commonMain` now owns the app root composable and route types.
A simple `KoloRoute` hierarchy provides typed Navigation 3 destinations.
The root uses explicit serializer registration so the back stack is valid on iOS, not just Android.
iOS host

Host-side changes

The iOS host now imports `ComposeApp` and wraps a Kotlin `ComposeUIViewController` factory.
Xcode now has a `ComposeApp` framework build phase alongside the existing Kotlin framework phases.
The debug-menu presentation flow stays at the host edge for now instead of being folded into the app root.
Important boundary This cut does not mean SwiftUI disappears from the iOS app lifecycle. It means SwiftUI is no longer the owner of the product root UI. The host remains thin, while the app surface moves into shared Compose where a shared navigation model can actually exist.
Validation

What was proven

The Kotlin side is verified directly. Full Xcode execution is partially blocked by the sandbox.
Passed

Kotlin compile

`:composeApp:compileAndroidMain` and `:composeApp:compileKotlinIosSimulatorArm64` both pass with the new shared root and Navigation 3 code.

Passed

Android app assembly

`:androidApp:assembleDebug` passed during the first integrated compile after the Navigation 3 API corrections.

Blocked

Full Xcode build

The Xcode host reaches the new script phases, but sandbox restrictions block Gradle wrapper process behavior and CoreSimulator access before a full app build can complete.

What This Is Not

What still remains

This establishes the base. It does not finish the navigation architecture.
Not done yet

Deep links and restoration policy

There is now a real typed back stack, but external entry, guard handling, safe-root behavior, and cross-feature route ownership are not yet formalized.

Not done yet

Module extraction

The first Navigation 3 runtime cut lives in `composeApp` to make the platform move real quickly. The next architectural step is still to extract the route contracts and app-owned navigation seam into dedicated navigation modules.

This page is the owner-facing summary for the iOS Compose + Navigation 3 platform cut. The detailed implementation still lives in code and the Markdown planning/workstream docs.
