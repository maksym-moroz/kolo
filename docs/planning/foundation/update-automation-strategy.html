Kolo · Update Automation Strategy

Automate version drift, but do it as part of a broader release integrity system.

The best setup for Kolo is not “just turn on a dependency bot.” It is a stack: one PR engine for updates, GitHub-native review and alerting, Gradle-level verification, and a staged rollout so the repo can scale without drowning in noisy upgrade work.

Decision: use Renovate as the primary updater for Kolo
Current Repo Readiness

What Kolo already has vs what is still missing

This is based on the current repo state, not generic advice.
Already in place

Strong baseline for automation

  • Central version catalog in gradle/libs.versions.toml.
  • Shared Gradle wiring in build-logic, which gives one stable update surface.
  • Aggregate CI entry points: qualityCheck, dependencyHealth, and health builds.
  • GitHub Actions already split quality, dependency analysis, and health concerns into separate workflows.
Missing today

Automation and integrity gaps

  • No update bot config is checked in yet.
  • No Gradle dependency lockfiles are checked in.
  • No gradle/verification-metadata.xml exists for dependency verification.
  • No explicit dependency-review gate, dependency-submission workflow, or SBOM/provenance flow exists yet.
  • The Gradle Wrapper uses validateDistributionUrl=true, but does not yet pin distributionSha256Sum.
Tool Decision

Which updater should own the repo

One PR engine is cleaner than overlapping bots that raise duplicate work.
Option Where it fits Why it is or is not the best primary choice
Renovate KMP / Android / server repos with Gradle, version catalogs, GitHub Actions, wrapper upgrades, and custom version files. Best fit for Kolo because it can cover Gradle, the wrapper, GitHub Actions, and custom version strings in one system, while giving strong grouping, scheduling, package rules, dashboards, and custom managers.
Dependabot Teams that want the simplest possible GitHub-native setup and can live with less policy control. Good default for simpler repos. For Kolo it is weaker as the single source of update automation because the repo spans Gradle catalogs, wrapper, build logic, and broader workflow hygiene. Keep GitHub security features even if Renovate owns update PRs.
Gradle-only discovery tools Manual visibility, local audits, or CI reporting. Useful as supplements, not as the main operating model. They tell you updates exist, but they do not provide the full PR workflow, cross-ecosystem coverage, scheduling, or policy rails needed for a scaling product repo.
Recommended Stack

What “good” looks like for a fast-scaling product repo

This is broader than dependencies alone because supply-chain drift is a platform problem.
1. Update PR engine

Renovate

Use Renovate as the single bot that raises version PRs for Gradle dependencies, plugins, the Gradle Wrapper, and GitHub Actions. Do not run Dependabot version PRs in parallel unless there is a very specific split of responsibility.

2. Security graph

GitHub-native alerts and review

Keep GitHub dependency graph, alerts, and dependency review in place regardless of which bot raises update PRs. These are repository safety rails, not bot competitors.

3. Build integrity

Gradle verification

Enable dependency verification and add wrapper checksum pinning. This is the line between “we upgraded” and “we know exactly which artifacts we trust.”

4. Reproducibility

Selective locking

Introduce Gradle dependency locking where it protects release-critical or unstable resolution paths. For a KMP repo, start selectively rather than locking every configuration on day one.

5. Workflow hardening

Immutable CI inputs

Pin GitHub Actions more strictly over time, prefer immutable references for critical workflows, and keep wrapper validation on every build path.

6. Release integrity

SBOM, attestations, immutable releases

As Kolo starts shipping real products, add dependency submission or SBOM generation, artifact attestations, and immutable releases so releases are traceable and auditable.

Operating Model

Best practices that matter more than the specific bot

The failure mode is usually process shape, not tool choice.
PR policy

Separate safe churn from planned churn

  • Auto-merge only low-risk patch and small minor updates after full CI.
  • Group related updates: Kotlin + Compose + KSP, AndroidX families, Ktor family, GitHub Actions.
  • Keep majors manual and explicitly owned.
  • Use schedules so upgrades land when the team is available to react.
Ownership

Every update lane needs a human owner

  • Assign CODEOWNERS or labels for Android, build logic, server, and workflow updates.
  • Use dashboards or issue views to track blocked majors.
  • Do not let “someday” majors silently pile up for quarters.
  • Treat upgrade debt like backlog, not background noise.
Testing

Gate on the same commands people trust locally

  • Reuse ./gradlew qualityCheck and the existing health build path as the merge gate for update PRs.
  • Keep update PRs small enough that failures are attributable.
  • Add a fast path for safe auto-merge lanes and a fuller path for risky groups.
Release safety

Provenance and rollback are part of scaling

  • Know which artifact was built from which commit and dependency graph.
  • Prefer immutable releases once published artifacts matter.
  • Keep rollback cheap by avoiding giant batched upgrade trains.
Kolo Rollout

The practical order for this repo

This sequence aims for real leverage early without loading the repo with ceremony.
Phase 1

Now

  • Add Renovate config.
  • Group update lanes by risk.
  • Add dependency review on pull requests.
  • Keep GitHub dependency graph and alerts enabled.
Phase 2

Next

  • Add Gradle dependency verification metadata.
  • Pin the Gradle distribution checksum in the wrapper properties.
  • Add explicit dependency submission if GitHub graph visibility is incomplete or too implicit.
Phase 3

Later

  • Add selective dependency locking for release-critical paths.
  • Harden GitHub Actions pinning policy.
  • Add SBOM, attestations, and immutable release posture when product shipping risk justifies it.
Non-Goals

What not to do

Most noisy automation programs fail here.
Avoid

Bot overlap and giant churn

  • Do not let Renovate and Dependabot both raise version PRs for the same surfaces.
  • Do not auto-merge major updates.
  • Do not batch unrelated risky upgrades into one “maintenance” PR.
  • Do not turn on every hardening feature in one step before the repo learns the workflow.
Remember

Scaling speed comes from boring discipline

  • Central version control is necessary, but not sufficient.
  • Reproducibility and provenance matter once more than one product depends on the baseline.
  • The upgrade program must stay legible to humans, not just “fully automated.”
