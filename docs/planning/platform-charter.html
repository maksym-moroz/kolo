Kolo · Platform Charter

Kolo exists to launch new mobile apps quickly with consistent quality.

This repo is a platform baseline, not a single app codebase with reusable leftovers. The charter exists to keep the baseline honest when multiple apps, shells, and agents start competing for space.

Mission

What Kolo is optimizing for

Launch velocity

Cheap and repeatable app creation

New apps should be faster to launch because they inherit architecture, tooling, quality rails, and shell conventions instead of rebuilding them.

Quality

Testability and observability are baseline features

Reliability is part of the platform, not a later cleanup. Every app should inherit deterministic boundaries, validation paths, and observability hooks.

Ownership

What Kolo owns vs what shells own

Kolo owns

Platform baseline

  • Build logic and repo-wide quality rails
  • Shared DI, store, persistence, navigation, and runtime-config contracts
  • Reusable capability modules
  • Observability and test infrastructure
  • Shell conventions and app-definition contracts
App shells own

App-specific identity

  • App ids and bundle ids
  • Branding, naming, icons, and market metadata
  • Enabled capabilities
  • App-specific integrations and monetization
  • Market- or client-specific composition
Guardrails

What must stay out

Do not add

Baseline pollution

  • App-specific copy with no reuse value
  • Partner-specific or one-off integrations in shared modules
  • Shell-specific hacks hidden as baseline abstractions
  • Broad feature logic added to shared without a named destination
Prefer instead

Extension rule

  • App-definition config
  • Capability composition
  • Adapter interfaces at the shell edge
