Kolo · Starter Architecture

The current architecture baseline for the platform and reference app.

The core rule is simple: keep shared Kotlin responsible for domain, data, state, and platform abstractions, while app shells stay thin at the edge. The repo already has a real store and runtime-config baseline, but feature extraction and platform decomposition are still in progress.

Current Reality

What is already true

This is implemented state, not just future structure.
Hosts

Reference app shells

  • apps/reference/androidApp is the current Android host for :androidApp.
  • apps/reference/iosApp is the current iOS host shell.
  • composeApp is a shared UI library, not the Android app module.
Core baseline

Already extracted

  • shared:core:store:api / impl is the real UDF/store baseline.
  • shared:core:appshell now holds the app-shell identity and capability contract.
  • shared:core:config:api / impl is the real runtime-config baseline.
  • debugmenu is isolated as internal tooling, not product UI.
Architectural Principles

What the repo is optimizing for

These are the stable rules behind implementation choices.
Shared ownership

Business rules stay shared

Shared code owns business rules, persistence contracts, networking contracts, and unidirectional data flow.

Thin edge

Hosts stay shallow

ViewModel and app hosts are adapters and lifecycle edges, not the center of product logic.

Platform safety

Platform quirks stay isolated

Navigation, notifications, deep links, and platform integrations should sit behind thin boundaries rather than leaking through common code.

Direction

Where the module tree is going

This is target direction, not fully implemented reality.
Shells

Thin app hosts

  • apps/reference/androidApp and apps/reference/iosApp prove the current baseline.
  • Future app shells should live under apps/<app-id>.
  • Shells should own branding, app ids, capability selection, and app-specific integrations.
Shared modules

Core plus capability split

  • shared:core:model, common, database, preferences, network, store, and config.
  • shared:feature:tasks, journal, reminders, and settings.
  • The broad shared module should keep shrinking over time.
Stack Baseline

Deliberately boring choices

The goal is stability and predictable launch velocity.
Language / build

Java 17 · Kotlin 2.3.20 · AGP 9.1.x

Build conventions live in build-logic and should remain the source of truth.

UI / DI

Compose MP · Metro

Compose Multiplatform is the main shared UI direction, with Metro as the DI baseline.

Persistence

Room + DataStore

Room KMP is intended for app data; DataStore KMP is intended for settings and runtime-config overrides.
