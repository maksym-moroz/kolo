Kolo · App Factory Workflow

The current first-cut workflow for launching a new app shell.

This is the working creation path for new apps built on top of the Kolo baseline. It is intentionally manual and explicit for now; the rule is to prove the shell contract on real apps before automating it away.

Workflow

Launch a new shell in eight steps

1

Create the shell

Create a new shell under apps/<app-id>/.

2

Add app-definition metadata

Create app-definition.yaml and name ids, display metadata, capabilities, runtime-config profile, and observability configuration.

3

Copy or template the hosts

Use apps/reference/ as the truth source for Android and iOS host structure.

4

Rename identities

Rename package ids, bundle ids, display names, and app-facing metadata.

5

Select capabilities

Decide which shared features the shell includes and how they are composed.

6

Bind app-specific integrations

Keep app-specific adapters at the shell edge instead of pushing them into shared modules.

7

Wire observability and environment config

Make sure logs, crash reporting, analytics, and runtime-config selection are set for the new app.

8

Run baseline validation

Prove the shell inherits baseline quality before treating it as a real launch candidate.

Validation

What the first shell must prove

  • Android host assembles
  • Shared Kotlin compiles for iOS
  • Capability composition is valid
  • Smoke flows run for the chosen feature set
Reference Rule

Do not automate too early

apps/reference/ stays as the truth source for shell structure until the contract has survived at least one additional real app. Do not jump to generators or scaffolding systems until the manual shape has proven stable.
