Kolo · App Shell Contract

An app shell is a thin host around the shared Kolo platform.

The shell should compose capabilities, own app-specific identity, and stay out of shared infrastructure concerns. This contract exists to keep shells shallow and reusable platform code honest.

Required Surface

What every shell must provide

Metadata

App definition

  • id
  • displayName
  • packageName
  • bundleId
  • Enabled capabilities
  • Runtime-config profile
  • Observability configuration
  • Release environment list
Hosts

Thin app edges

  • Android host wiring
  • iOS host wiring
  • Branding assets
  • Release identity
  • App-specific integration bindings
Boundary

What shells may own vs must not own

Shell may own

Identity and composition

  • Branding and copy
  • Capability inclusion or exclusion
  • App-specific adapters
  • Market-specific integration choices
Shell must not own

Shared platform concerns

  • Shared store primitives
  • Shared navigation contracts
  • Shared persistence abstractions
  • Duplicated platform infrastructure
Composition Rule

How shells and platform relate

Opt-in capabilities

The shell decides what to mount

Capabilities should be opt-in from the shell. The shell chooses what to include; the platform decides how shared contracts and infrastructure behave. The current truth source for shell wiring is apps/reference/.
