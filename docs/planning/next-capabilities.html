Kolo · Next Core Infra

The next capability set we are building before real app and MVP work.

This is the next owner-facing capability list for Kolo as a reusable platform baseline. These are not end-user product features like tasks or journaling. They are the shared core capabilities the baseline needs so future apps do not feel underpowered, improvised, or too shell-specific.

Important distinction

Infra first, product capability second

Tasks, journals, reminders, and settings screens are shared product capabilities built on top of core infra. The list below focuses on the infra layer first: the contracts, seams, and runtime rails that make those future capabilities portable across apps.

Not in this wave

Billing is intentionally not first-class yet

Billing may become a reusable capability later, but it is not yet baseline core infra. Unless most future apps monetize in the same way, billing should stay behind the first foundation wave instead of shaping the platform too early.

Full Ordered List

The next ten core capabilities

Each item is written as a reusable baseline capability, not as a one-off feature for the reference app.
01

App definition and capability composition

The platform needs a real contract for what an app is: metadata, environment profile, enabled capabilities, release identity, and composition rules.

Lets future shells opt into capabilities intentionally instead of inheriting everything by accident.
Creates the seam other shared infra can depend on.
02

Shell bootstrap and DI/runtime ownership

Android and iOS hosts should follow one predictable startup model for graph creation, lifecycle boundaries, and app-owned platform bindings.

Prevents each future shell from inventing its own wiring style.
Keeps the shared layer honest about what it owns versus what only the shell can own.
03

Persistence and migration contract

Room/DataStore ownership, repository boundaries, stable ids, schema export, and migration testing need to be part of the baseline before feature slices spread.

This is where “durable storage shape” becomes real instead of implied.
Later tasks, reminders, journals, and settings should plug into this rather than redefine it.
04

Navigation, deep links, and restoration

Typed routes, modal rules, back behavior, deep-link parsing, and restoration policy should be infrastructure, not screen-by-screen improvisation.

Without this, every MVP flow ends up teaching the shell a different navigation language.
This is a portability requirement for future apps, not just a UI convenience.
05

Runtime config, feature flags, and dev menu

Environment switching, local overrides, feature toggles, and safe internal tooling make the baseline debuggable and rollout-ready.

The repo already has a first cut here, but it should be treated as one of the main platform capabilities, not a side utility.
This is one of the biggest force multipliers for development speed.
06

Settings and preferences framework

A shared way to model user preferences, default values, persistence, and shell-facing configuration surfaces.

Small on paper, but a baseline feels incomplete without it.
Should stay distinct from runtime config and from end-user feature state.
07

Notifications and scheduling abstraction

Local and push notification contracts, permission flow seams, reminder delivery boundaries, and scheduler ownership on Android and iOS.

Critical for any product family where reminders are part of the shape.
Should separate stored intent from platform delivery mechanics.
08

Background work and offline job orchestration

A standard place for deferred work, retries, cleanup, refresh jobs, and future sync triggers instead of scattering platform schedulers across features.

Even before full sync exists, the baseline needs a home for background behavior.
Protects the shared platform from “just launch a worker here” entropy.
09

Observability

Structured logging, analytics seams, crash reporting, performance traces, and diagnostics hooks should ship as platform defaults.

A reusable baseline should be measurable and debuggable in production by default.
App shells can provide app-specific keys and policies at the edge.
10

Release policy rails

Force-update/version gating, release channels, rollout safety checks, smoke validation, and release performance baselines.

This is where the baseline starts feeling “ship-ready” rather than just architecturally tidy.
Force update belongs here as a release-policy concern, not as a one-off screen feature.
Practical reading of this list The first implementation wave does not mean building all ten at once. It means starting with the composition seam, then the runtime/bootstrap seams, then the capabilities that make the baseline feel like a real app platform: config, settings, notifications, observability, and release policy.
Boundary Rule

What this list is not

This keeps the platform from collapsing back into “everything is shared infra.”
Not core infra

Tasks, journals, reminders

These are shared product capabilities that should be built on top of the infra listed here, not confused with the infra itself.

Not first-wave baseline

Billing

Billing only becomes baseline when it is clearly common across the future app family; otherwise it stays a later capability.

Not shell-owned

Core contracts

App shells should choose capabilities and supply app-specific bindings, but the platform should own how these shared contracts behave.
